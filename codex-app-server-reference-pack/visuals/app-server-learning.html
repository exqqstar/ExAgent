真实链路图

这个页面用一个真实请求跑通 Codex app-server：从启动、initialize、thread/start、 turn/start，到 core 事件流回客户端。每张图右侧写当前节点发生了什么、携带什么参数、 对应代码入口在哪里。

场景 
Client: VS Code / Desktop / local controller
Transport: stdio://
cwd: /Users/enxiangqiu/Documents/New project 2
Request: "给我讲讲 app-server"
01

启动 app-server

CLI 只负责把启动参数传进来。真正的 runtime 在 run_main_with_transport_options 里构造：环境、配置、认证、数据库、transport 和消息处理器。

启动阶段：从 CLI 到 MessageProcessor 
flowchart TD
  A["cli/main.rs
Subcommand::AppServer"] --> B["run_main_with_transport(...)
传入启动参数"]
  B --> C["run_main_with_transport_options(...)
构造 app-server runtime"]
  C --> D["EnvironmentManager::new
local / remote execution environments"]
  C --> E["channels
transport_event / outgoing / outbound_control"]
  C --> F["ConfigManager::new
config.toml + CLI override + request override"]
  F --> G["load_latest_config
得到基础 Config"]
  C --> H["AuthManager + state db
认证、SQLite、tracing、analytics"]
  C --> I{"transport"}
  I --> J["Stdio
stdin/stdout JSONL"]
  I --> K["UnixSocket
本机控制 socket"]
  I --> L["WebSocket
实验性 ws://IP:PORT"]
  I --> M["Off
不开放本地入口"]
  C --> N["outbound router task
把消息发给正确连接"]
  C --> O["processor task
创建 MessageProcessor"]
  O --> P["MessageProcessor::new
内部创建 ThreadManager"]

这里发生了什么

入口参数

arg0_paths 给 core 找当前 codex 可执行文件和 sandbox helper； cli_config_overrides 是 -c key=value； transport 决定连接方式。

EnvironmentManager

后续 shell、文件系统和 sandbox 都需要知道在哪个 environment 里执行。

EnvironmentManagerArgs::new(ExecServerRuntimePaths::from_optional_paths(...))
ConfigManager

统一把全局 config、项目 config、CLI override、request override 合成最终 Config。

MessageProcessor

连接和请求的总处理器。它会创建 ThreadManager，后面 thread/turn 都从这里进 core。

02

initialize 握手

每个连接必须先 initialize。app-server 在连接级别记录客户端身份、实验 API 开关、 通知过滤偏好，然后才允许后续 request 进入业务分发。

initialize：连接身份和能力边界 
flowchart TD
  A["Client sends JSON-RPC
method: initialize
id: 1"] --> B["TransportEvent::IncomingMessage"]
  B --> C["MessageProcessor::process_request"]
  C --> D["serde_json -> ClientRequest::Initialize"]
  D --> E["session.initialize(...)
OnceLock 记录连接状态"]
  E --> F["set_default_originator
clientInfo.name -> upstream metadata"]
  E --> G["USER_AGENT_SUFFIX
client name + version"]
  E --> H["analytics track_initialize"]
  E --> I["InitializeResponse
userAgent / codexHome / platform"]
  I --> J["OutgoingMessageSender::send_response"]
  J --> K["Client receives response"]

真实参数长这样

request

clientInfo.name 标识调用方，experimentalApi 决定能否调用实验 API。

{"id":1,"method":"initialize","params":{"clientInfo":{"name":"codex_vscode","version":"0.1.0"},"capabilities":{"experimentalApi":true}}}
连接状态

状态存在 ConnectionSessionState，未初始化连接调用业务 API 会返回 Not initialized。

通知过滤

optOutNotificationMethods 让客户端拒收某些 notification，避免旧客户端处理不了新事件。

03

thread/start 创建 core session

thread/start 把外部 API 参数变成 ConfigOverrides， 加载最终 Config，再调用 ThreadManager 创建 core thread。

thread/start：从 API 参数到 core thread 
flowchart TD
  A["ClientRequest::ThreadStart
id: 2"] --> B["CodexMessageProcessor::thread_start"]
  B --> C["解构 ThreadStartParams
model / cwd / sandbox / permissions / instructions"]
  C --> D["build_thread_config_overrides
params -> ConfigOverrides"]
  D --> E["thread_start_task
后台任务中执行"]
  E --> F["ConfigManager::load_with_overrides
得到最终 Config"]
  F --> G["validate environments
validate dynamic tools"]
  G --> H["ThreadManager::start_thread_with_options"]
  H --> I["core Codex::spawn"]
  I --> J["Session::new
加载 model / skills / plugins / AGENTS.md / exec policy"]
  J --> K["spawn submission_loop
等待 core Op"]
  H --> L["NewThread
thread_id + CodexThread + session_configured"]
  L --> M["ensure_conversation_listener_task
当前连接订阅 thread events"]
  M --> N["ThreadStartResponse"]
  M --> O["ThreadStarted notification"]

这个请求携带什么

request

本例只给 cwd 和 model，其他都走配置默认值。

{"id":2,"method":"thread/start","params":{"cwd":"/Users/enxiangqiu/Documents/New project 2","model":"gpt-5.2-codex"}}
ConfigOverrides

覆盖项不会直接跑 agent，而是先参与构造最终 Config。

model, model_provider, cwd, approval_policy, sandbox_mode, permissions, base_instructions...
ThreadManager

它管理 core thread 生命周期：start、resume、fork、get、shutdown。

listener

thread 创建后马上 attach listener，否则客户端收不到后续模型和工具事件。

04

turn/start 提交用户输入

turn/start 不直接等模型回答。它把用户输入转换成 core input， 构造 core Op，塞进 core 的 submission queue，然后立刻返回 InProgress。

turn/start：外部 input 到 core Op 
flowchart TD
  A["ClientRequest::TurnStart
id: 3"] --> B["CodexMessageProcessor::turn_start"]
  B --> C["validate_v2_input_limit
输入大小校验"]
  C --> D["load_thread(threadId)
找到 CodexThread"]
  D --> E["V2UserInput::into_core
wire input -> core input"]
  E --> F["build TurnContext
thread defaults + turn overrides"]
  F --> G{"是否有 turn overrides"}
  G -->|yes| H["validate_turn_context_overrides
让 core 先验证配置变更"]
  H --> I["Op::UserInputWithTurnContext
items + TurnContext updates"]
  G -->|no| J["Op::UserInput
items + inherited context"]
  I --> K["submit_core_op"]
  J --> K
  K --> L["CodexThread::submit_with_trace"]
  L --> M["Codex::submit_with_trace
生成 submission id / turn id"]
  M --> N["tx_sub.send(Submission)
Submission 包含 Op"]
  N --> O["TurnStartResponse
status: InProgress"]
  N --> P["core submission_loop
handlers -> new_turn_with_sub_id -> run_turn"]

真实 input

request

外部 API 用 codex_app_server_protocol::UserInput 表示输入。

{"id":3,"method":"turn/start","params":{"threadId":"019a-example","input":[{"type":"text","text":"给我讲讲 app-server","textElements":[]}]}}
into_core

把 API wire 类型转成 core 内部类型。现在字段很像，但这层隔离让外部 API 和 core 可以分别演进。

validate overrides

如果这次 turn 临时换 cwd、model、permissions，先问 core 这些更新是否允许。

submit_with_trace

生成 turn id，附带 trace，把 Op 发进 core queue。成功只代表排队成功。

05

事件流返回客户端

core 真正执行 turn 后会不断发 EventMsg。 app-server listener 读取这些事件，更新 thread 状态，再翻译成 JSON-RPC notification 推给客户端。

core EventMsg -> app-server notification 
flowchart TD
  A["core run_turn
模型调用、工具调用、命令执行"] --> B["Session sends EventMsg
TurnStarted / AgentDelta / Tool / Exec / TurnComplete"]
  B --> C["CodexThread::next_event"]
  C --> D["listener task
ensure_listener_task_running_task"]
  D --> E["ThreadState::track_current_turn_event
更新 active turn 摘要"]
  D --> F["ThreadStateManager::subscribed_connection_ids
找到订阅连接"]
  E --> G["apply_bespoke_event_handling
EventMsg -> ServerNotification"]
  F --> G
  G --> H["ThreadScopedOutgoingMessageSender"]
  H --> I["OutgoingMessageSender
ToConnection / Broadcast"]
  I --> J["Client UI
流式文本、工具状态、完成事件"]

为什么需要 listener

turn/start response 很短

它只返回 TurnStartResponse { status: InProgress }，不包含最终答案。

模型输出是异步流

文本 delta、reasoning、工具调用、shell 输出、文件变更、错误和完成都通过 notification 发回。

状态管理

ThreadStateManager 维护 connection 和 thread 的订阅关系；ThreadState 维护当前 turn 摘要。
